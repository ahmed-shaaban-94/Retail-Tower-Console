## What this directory contains (and does not contain)

This directory documents the **consumption boundary** between
Retail-Tower-Console (this repo) and Data-Pulse-2 (the upstream backend
contract authority). It is a **policy document**, not a contract source.

### What lives here

- This `README.md` — consumption boundary policy
- One Markdown file per route family **whose api-readiness row(s) have
  been verified to `draft` or `stable`**. At foundation time, that
  means:
  - `rf1-auth-context.md` — RF-1 consumption boundary (verified
    2026-05-25)
- One Markdown file per *future* per-family slice. RF-4a's file will
  land when `007-rf4a-unknown-items` re-verifies the rows. RF-2 / RF-3
  / RF-5 / RF-6 / RF-7 files land as their api-readiness rows resolve.

### What does NOT live here (Principle 2)

- ❌ OpenAPI source files (Data-Pulse-2 owns those)
- ❌ Copied or paraphrased schema definitions
- ❌ Generated TypeScript / JavaScript client code (Principle 8 — the
  client is generated by the scaffold slice and stored where that
  slice decides)
- ❌ Mock servers, MSW handlers, or any executable contract artifact
  (those belong in test directories created by per-family slices,
  under per-family approval)

## Boundary rules

### Rule 1 — References, not content

Every contract file under this directory cites Data-Pulse-2 by **path +
commit-SHA + operation ID**. It does NOT inline schema content.

Example of what's permitted:

> The active-context endpoint is `GET /api/v1/context/me` with
> `operationId: getActiveContext` in
> `Data-Pulse-2/packages/contracts/openapi/context.openapi.yaml`
> @ `main` SHA `b5142fe`. The response shape is `ContextResponse` (see
> the generated client when slice `002-tooling-and-scaffold` ships).

Example of what's forbidden:

> ❌ The active-context endpoint returns `{ user: UserSummary,
> active_tenant: TenantSummary | null, active_store: StoreSummary |
> null, active_role_code: string | null, memberships:
> MembershipSummary[] }`.

The second form **inlines schema content** and creates a drift surface
the constitution is designed to prevent. The first form names the
upstream source so the generated client is the only place the schema
lives in this repo.

### Rule 2 — Generated client is the only API surface

Once slice `002-tooling-and-scaffold` ships a generated OpenAPI client,
every per-family slice MUST import from that client. Hand-rolled fetch
calls to Data-Pulse-2 endpoints are forbidden (Principle 8).

The generated client's storage location, refresh cadence, and version
pinning are decided in slice `002`. This README does not pre-empt
that decision.

### Rule 3 — No POS-direction calls

The console MAY:

- Read POS-originated data via Data-Pulse-2 (e.g., audit events,
  unknown items).

The console MAY NOT:

- Call a POS device directly.
- Call a Data-Pulse-2 endpoint that writes data back to a POS device.
  (If such an endpoint exists, it's POS-Pulse-bound and outside the
  console's scope — Principle 3 / FR-003.)

### Rule 4 — Backend enforces authorization

Every API call the console makes assumes the backend will check whether
the actor is allowed. The console does NOT pre-check, does NOT branch
on "the user is a Platform Admin so they can call this," and does NOT
hide endpoints to enforce scope (Principle 7 / FR-002).

The console MAY hide UI controls based on the backend's response shape
(e.g., if the backend returns `is_platform_admin: false`, the console
can hide the "Manage tenants" link — but the link being hidden is a
*display* decision, not an authorization decision; the backend will
still reject the call if it's made anyway, which is the actual
authorization boundary).

### Rule 5 — Cookie-based session (no token plumbing)

Data-Pulse-2 uses an `HttpOnly + Secure + SameSite=Lax` session cookie
named `dp2_session` (api-readiness.md §RF-1 verification). The console
does NOT:

- Read the cookie from JavaScript (`HttpOnly` prevents this anyway).
- Attach `Authorization: Bearer ...` headers to API calls (the cookie
  travels automatically on same-site requests).
- Persist any auth artifact in `localStorage`, `sessionStorage`, or
  IndexedDB.

The browser handles auth transport. The console's only "auth code"
is: call `signIn` → handle 200 → call `getActiveContext`. Sign-out is
`signOut` → handle 204 → navigate to sign-in.

### Rule 6 — Refresh strategy

`POST /api/v1/auth/refresh` (operationId `refreshSession`) extends the
sliding session window. The console MAY call it proactively to avoid
expiry mid-action. The cadence and trigger (e.g., on tab focus, on
last-activity-was-N-minutes-ago) are decided in slice
`003-rf1-auth-shell`.

### Rule 7 — Error rendering policy

Every API error response carries `{ error: { code, message, request_id } }`.

| Status | Console behavior |
| --- | --- |
| 200 / 201 / 204 | Render success / success state |
| 400 | Surface `message` in the UI; surface `request_id` in the error footer / support panel |
| 401 | Navigate to RF-1 sign-in (do NOT show a "your session expired" toast and then stay on the protected page — the page would render with stale data) |
| 403 | Render a generic "you don't have access" with `request_id`; do NOT explain *why* (Principle 7) |
| 404 | Render "not found" identically regardless of cause (per `context.openapi.yaml` FR-ISO-4) |
| 409 (e.g., "no active tenant set") | Surface the precondition error; offer the appropriate fix-up navigation (e.g., open the tenant chooser) |
| 429 | Surface a retry-after notice |
| 5xx | Render a generic error with `request_id`; offer a "report a problem" path that includes the `request_id` |

Concrete component-level shapes are deferred to slice
`003-rf1-auth-shell` and per-family slices.
